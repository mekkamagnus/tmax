# Unified Observability for the `*Messages*` surface

Today a user presses a key, sees a message flash in the status line, and
once it disappears there is no record of what happened. Shell commands
silently drop their stderr and exit code. Async subprocesses pipe stderr
but never read it. Test runs report only a summary string while their
structured results sit in a store the log never sees. After a daemon
restart, every one of these is gone. This plan fixes all four with one
unified `LogEntry` schema, one ring store, five category-filtered
virtual buffers, and JSONL persistence — with a mirror rule that keeps
`*Messages*` as the one buffer a user can always trust to show failures
and warnings.

The hard-to-reverse bet is decided up front: **one schema, many
buffers**. Once the schema and the `messages` view predicate are
shipped, every call site and every persisted line depends on them. We
commit to that shape here and scope the first cut to prove it without
foreclosing the deferred RFC-017 agent-activity log.

## What changes concretely

A user runs `(shell-exec "bun test")` from the editor. Today the
`exitCode`, `durationMs`, and stderr are captured only in the return
value and lost to the log. After this plan:

- `*Shell Output*` shows `[2026-06-17 14:32:07] [info] [shell] bun test [exit 1] [340ms]` plus the last ~4 KB of stdout+stderr.
- Because the exit was non-zero (level `error`), `*Messages*` mirrors that line automatically — the buffer the user already checks shows the failure without being polluted by successful runs.
- After `tmax --stop` and a restart, the entry is still there: it was appended to `~/.config/tmax/messages.log` as JSONL and tail-loaded into the ring on startup.

Same shape holds for a crashing `make-process`, a failing trt suite, a
workspace auto-save failure, and the unbound keys in normal mode (today
the one handler that never logs).

## Objective and done-state

Done means: **no editor event, shell command, subprocess, test run, or
auto-save can vanish**. Every one lands in a category buffer with full
context, every `warn`/`error` mirrors into `*Messages*`, and the whole
history survives a restart. Verified by: unbound key in normal mode
appears in `*Messages*`; `(shell-exec "false")` mirrors an error;
`make-process "sh -c 'exit 2'"` records `exitCode: 2`; a failing trt
run lands in `*Tests*`; killing and restarting the daemon preserves
the prior session's entries.

## Scope

**In:** unified `LogEntry` schema + single `Log` ring store with
append-based (lazy) render; replace dual `messageLog`/`daemonLog` rings;
fill the `normal-handler.ts` logging gap and elevate unbound keys to
`warn` across all handlers; two-tier `(message)`/`(echo)` split;
richer entries (`frameId`, full-date timestamps, program-run fields);
capture `shell-command`/`shell-exec`, `make-process` start+exit+stderr,
and trt runs; three new virtual buffers (`*Shell Output*`, `*Async
Output*`, `*Tests*`); JSONL persistence to `~/.config/tmax/messages.log`
with 5 MB rotation and startup tail-load; daemon `query` and T-Lisp
`(log-query)`/`(observability-buffer)` API.

**Out (deferred):** full RFC-017 per-RPC agent-activity log (the lazy
render + category routing here are its stated prerequisites); SPEC-053
`*Test Explorer*` UI (layers on the `*Tests*` buffer); per-frame log
level (one global `minLevel` for now).

The five virtual buffers are **views over the same store**, not separate
rings. The `messages` view is special — it is the union of
`category ∈ {editor, autosave}` with every `warn`/`error` entry from any
category (the mirror rule). That rule is what makes the multi-buffer
split safe: a failed shell command or crashed subprocess surfaces in the
buffer the user already checks, while its full output lives in its own
buffer so it cannot drown editor events. Daemon connection events stay
isolated in `*daemon*` because they are `info` level — they do not
trigger the mirror.

## The unified schema

```data-model
LogCategory = 'editor' | 'daemon' | 'shell' | 'process' | 'test' | 'autosave'
LogLevel    = 'debug' | 'info' | 'warn' | 'error'

LogEntry {
  ts          Long       // epoch ms — full date, unambiguous across sessions
  level       LogLevel   // severity; warn|error trigger the messages-view mirror
  category    LogCategory
  text        String     // human-readable summary line
  command     String?    // T-Lisp command name (existing, on errors)
  frameId     String?    // which client/frame caused it (D2 attribution)
  exitCode    Int?       // shell/process/test only
  durationMs  Int?       // shell/process/test only
  outputTail  String?    // last ~4 KB of stdout+stderr, capped
  pid         Int?       // make-process only
}
```

This is the hard-to-reverse decision. Every producer writes this shape;
every consumer (buffer view, JSONL line, daemon query, `(log-query)`)
reads it. Optional fields are absent on entries where they don't apply,
so the schema scales from a one-line editor message to a full
program-run record without overloading.

<callout tone="decision">
**Mirror rule — alpha-stage high observability.** Every `warn`/`error`
entry from any category mirrors into `*Messages*`. Unbound keys are
deliberately elevated from SPEC-016's `debug` to `warn` so they mirror
automatically — "I pressed a key and nothing happened" is the exact
confusion this log exists to dispel. This can be relaxed back to `debug`
post-alpha. The SPEC-047 `*daemon*` split is preserved because
connection events are `info`, not `warn`/`error`.
</callout>

## Roadmap

Four phases, ordered by dependency. Phase 1 is a pure refactor that
unblocks everything else by fixing the O(n) render blocker; Phase 4
depends on the schema being settled in Phase 1.

### Phase 1 — Foundation: unified schema + lazy-render store

Replace the dual `messageLog`/`daemonLog` rings with one `Log` store
and make `render()` O(1) amortized. No new user-facing behavior — every
existing `*Messages*`/`*daemon*` test must pass unchanged. This phase
**must land the lazy render** or Phase 3 regresses (it is the stated
RFC-017 deferral reason).

```diff
- private messageLog = new MessageLog();
- private daemonLog = new MessageLog();
+ private log = new Log();
```

The lazy render is the load-bearing mechanic:

### Phase 2 — Routing completeness + richer context

Fill the `normal-handler.ts` logging gap (the one handler with zero
`logMessage` calls — SPEC-016 Phase 2, implemented in 3 of 4 handlers).
Introduce the two-tier `(message)`/`(echo)` split so deliberately
transient messages (which-key hints, prompts, `C-w` prefixes) are
explicitly echo-only instead of vanishing. Elevate unbound keys from
`debug` to `warn` across all four handlers for the alpha mirror rule.

The T-Lisp routing rule is explicit and matches Emacs:

| Primitive | `statusMessage` | `Log` entry | Use for |
|---|---|---|---|
| `(message ...)` | yes | yes (`info`) | real editor messages — file ops, command results, summaries |
| `(log-message :LEVEL ...)` | yes (info+) | yes (explicit level) | leveled logging from T-Lisp |
| `(echo ...)` (NEW) | yes | **no** | deliberately transient — which-key hints, prompts, prefixes |

`(message)` keeps its current log+echo behavior (`tlisp-api.ts:686-687`)
unchanged. `(echo)` is the **new** escape hatch so call sites that today
set `statusMessage` directly without logging can be classified
explicitly instead of vanishing. The TS-side counterpart is
`setEchoOnly(text)` on `Editor`.

### Phase 3 — Program-run capture + new buffers

Wire `shell-command`/`shell-exec`, `make-process` (spawn + exit +
stderr), and trt runs into the store with their structured fields. Stand
up the three new virtual buffers as category-filtered renders. The
mirror rule surfaces failures in `*Messages*` automatically — no
per-call-site mirror code.

### Phase 4 — Persistence + daemon query + T-Lisp API

Append every entry to `~/.config/tmax/messages.log` as JSONL with 5 MB
rotation, tail-load the last `maxSize` entries on startup, flush on
graceful shutdown. Extend `query messages` with `category` and add
`query log`; add `(log-query :category :level :last)` and
`(observability-buffer NAME)`.

## Risks

<callout tone="warning">
**Lazy-render correctness is the whole bet.** If the dirty-set logic
misses a case (a view that should have been invalidated but wasn't), a
buffer silently shows stale content. The mirror rule makes this subtle:
a `shell` `error` entry must dirty `messages` even though its category
is `shell`. Mitigation: the `invalidate` method always adds `messages`
to the dirty set, and `getEntries({ view: 'messages' })` applies the
level/category predicate at read time — the cache key is the view name,
the cache invalidation is conservative. Unit test asserts cache hit on
repeat render with no write, and cache miss after a mirrored-category
write.
</callout>

<callout tone="warning">
**`make-process` stderr is currently piped but never read**
(`tlisp-api.ts:1158`). Adding a second async reader mirroring the stdout
reader is straightforward but unbounded — a chatty subprocess could fill
the 4 KB `outputTail` cap instantly and evict earlier editor entries
from the 1000-entry ring. Mitigation: the cap is on the *entry's stored
tail*, not the stream; the live stream still flows to the caller's
`:filter` unchanged. Ring eviction of a process start entry before its
exit entry is acceptable (both remain individually correct).
</callout>

<callout>
**Persistence write strategy.** Append-per-write is crash-safe (no lost
tail on crash) but adds an fsync-adjacent cost per event. Bun's buffered
writes keep this acceptable; the shutdown `flush()` is belt-and-suspenders.
If profiling later shows a hot path, the escape hatch is a periodic
flush every N ms — documented, not built now.
</callout>

## Verification

Beyond `bun run typecheck` and `bun test`, the plan changes daemon
state, file I/O, and cross-session behavior, so the end-to-end smoke
matters:

1. **Routing gap closed:** in a renderer test, send an unbound key in
   normal mode → assert `*Messages*` contains `[warn] Unbound key:`.
2. **Shell capture:** `tmax -e '(shell-exec "false")'` → switch to
   `*Shell Output*` (entry with `exitCode: 1`, duration, output tail) →
   switch to `*Messages*` (mirrored `error` line).
3. **Process capture:** `tmax -e '(make-process :command "sh -c \"exit 2\"")'`,
   wait → `*Async Output*` has exit entry `exitCode: 2`; `*Messages*`
   has the mirror.
4. **Test capture:** M-x `trt-run-tests` against a fixture with one
   failing test → `*Tests*` has the entry with failing name in
   `outputTail`; `*Messages*` has the summary.
5. **Persistence:** run step 2, `tmax --stop`, restart daemon → switch
   to `*Messages*` → the prior `error` line is still present
   (tail-loaded). Inspect `~/.config/tmax/messages.log` for the JSONL
   line.
6. **Corrupt-line resilience:** append a truncated line to the log file,
   restart → startup succeeds, the truncated line is skipped.
