Elm Architecture Data Flow

→
handleKey(key) → EditorState
→
view(state) → styled strings
→
Screen (ANSI or React)

The Elm Architecture pattern: input → update → view → screen. Both frontends share the same update logic (Editor.handleKey) and view logic (pure render functions). Only the screen layer differs.

Ink vs Steep Comparison

Ink (existing)

  • React reconciler + Fiber scheduler
  • <Box>, <Text>, useInput, useState
  • ~5–10ms render cycle per keystroke
  • JSX compilation required (.tsx)
  • Dependencies: ink, react
  • Component lifecycle management

Steep (new)

  • Direct ANSI escape writes
  • screen.ts, input.ts, style.ts (~170 lines total)
  • ~1–2ms render cycle per keystroke
  • Pure TypeScript (.ts)
  • Zero external dependencies
  • Functional update → view → paint loop

One-line change to editor.ts: add return this.state; at end of handleKey(). Signature changes from Promise<void> to Promise<EditorState>. Enables the Elm Architecture update(msg, model) → model pattern. No logic changes inside handlers or API ops.

The Editor still mutates state internally (~104 direct mutations in editor.ts, ~350 across api/*.ts). The return value is a bridge to functional patterns — not a full immutable rewrite.

Create SteepFrontend
The Elm Architecture runtime — the core loop:
const render = () => {
  const lines = renderBufferLines(state, width, height - 2);
  const status = renderStatusLine(state, width);
  screen.clear();
  lines.forEach((line, i) => screen.writeAt(i, 0, line));
  screen.writeAt(height - 1, 0, status);
};

input.onKey(async (msg) => {
  state = await editor.handleKey(msg.key); // ← Elm update
  render();                              // ← Elm view
});

Key Design Decisions

Why handleKey returns state
The Elm Architecture requires update(msg, model) → model. By returning the state, both frontends get a clean functional interface. The internal mutation pattern (stateAccess proxy) is an implementation detail that doesn't leak to the frontend layer.
Why not fully immutable Editor
That would require changing ~104 this.state. mutations in editor.ts plus ~350 across api/*.ts. That's a separate chore. The return-value change is the minimal bridge to enable Elm Architecture without rewriting the entire editor.
Why structured render segments
Shared render functions return {text: string, style: StyleFlags}[] — not ANSI strings. Ink maps to <Text> props; Steep maps to ANSI codes. Neither frontend parses the other's format.
Steep is intentionally minimal
~250 lines total (screen + input + style + index). No virtual DOM, no reconciler, no component lifecycle. Proves the direct-ANSI approach works and can be extended later with layout helpers, diffing, etc.
Terminal resize
Steep handles SIGWINCH directly in screen.ts. Ink handles it through React's useEffect + process.stdout 'resize' event. Both call editor.updateTerminalSize() and re-render.
